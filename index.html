与第一种的本质区别

第一种（你最初描述）
同一笔 TriggerSmartContract 里同时有 call_value + approve data。
钱包 UI 只展示 call_value。
第二种（本页面模拟）
payload 对象里存在多份表示（raw_data / raw_data_hex / nested transaction），
UI 读一份，签名模块读另一份。
如何在 imToken 里测试真实效果
1. 把本页面或第一个 on-chain 文件上传到可访问地址
2. 在 imToken App 内置浏览器中打开页面
3. 连接钱包后构造交易并签名，观察 imToken 的预览弹窗
注意：第二种的“不一致”主要发生在钱包解析 payload 的阶段，真实上链仍需构造合法的 TriggerSmartContract。
